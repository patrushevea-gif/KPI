# KPI Platform — быстрый старт и путь к масштабированию

## 1) Что вы хотите получить

Система должна уметь:
- принимать большой набор числовых полей (например, 1000+ значений);
- распределять значения по уровням:
  - **сотрудник** (личные KPI),
  - **отдел/функция** (агрегированные KPI подразделения),
  - **направление/завод** (дашборды верхнего уровня);
- хранить целевые значения KPI (план) и считать факт/выполнение;
- показывать данные в разрезе периода (день/неделя/месяц/квартал);
- позволять быстро начать, а затем масштабироваться без полной переделки.

---

## 2) Быстрый и простой запуск (MVP за 2–6 недель)

### Вариант A — **No-code/Low-code стек** (самый быстрый)

**Стек:**
- Источник данных: Bitrix24 + Excel/CSV + ручной ввод.
- Хранилище: Google Sheets / Airtable / Baserow.
- ETL/автоматизация: Make.com / n8n.
- Визуализация: Looker Studio / Power BI.

**Как работает:**
1. Все «сырые» поля (1000+) загружаются в таблицу `fact_events`.
2. Отдельные таблицы-справочники:
   - `employees` (сотрудники, отделы, роли),
   - `departments`,
   - `metrics` (описание метрики, единица измерения, формула),
   - `kpi_targets` (плановые значения KPI по периодам).
3. Скрипты/сценарии в n8n/Make считают агрегаты:
   - сотрудник → отдел → завод.
4. BI-дашборд читает готовые агрегаты.

**Плюсы:**
- минимальный порог входа;
- быстро увидеть результат;
- дешево на старте.

**Минусы:**
- тяжелее поддерживать сложную логику формул;
- контроль качества данных слабее, чем в полноценном DWH.

**Кому подходит:** если нужно быстро проверить гипотезу и показать рабочий прототип.

---

### Вариант B — **Легкий “data stack”** (оптимальный баланс)

**Стек:**
- PostgreSQL (единое хранилище),
- Python (ETL),
- dbt (трансформации),
- Metabase / Superset / Power BI (дашборды),
- GitHub (репозиторий логики и конфигурации KPI).

**Как работает:**
1. ETL забирает данные из Bitrix24/API/CSV.
2. Сырые данные пишутся в `raw_*` таблицы.
3. dbt строит модели:
   - `stg_*` (очистка и стандартизация),
   - `int_*` (промежуточные расчеты),
   - `mart_employee_kpi`,
   - `mart_department_kpi`,
   - `mart_dashboard_metrics`.
4. BI-инструмент показывает результаты.

**Плюсы:**
- прозрачная логика расчетов;
- версионирование в GitHub;
- проще масштабировать.

**Минусы:**
- чуть дольше запуск, чем no-code.

**Кому подходит:** если сразу нужна «взрослая» архитектура, но без перегруза.

---

## 3) Рекомендуемая модель данных (универсальная)

Независимо от стека, держите единую структуру:

- `fact_value`
  - `event_id`
  - `metric_id`
  - `employee_id` (nullable)
  - `department_id` (nullable)
  - `plant_id`
  - `value`
  - `event_date`
  - `source_system`

- `dim_metric`
  - `metric_id`
  - `metric_name`
  - `level` (`employee` / `department` / `plant`)
  - `aggregation_type` (`sum`, `avg`, `ratio`, `weighted`)
  - `direction` (`higher_is_better`, `lower_is_better`)
  - `weight`

- `dim_employee`
- `dim_department`
- `kpi_target`
  - цель KPI на период (месяц/квартал/год)

- `kpi_result`
  - рассчитанный факт,
  - `% выполнения`,
  - итоговый балл KPI.

---

## 4) Как связать «Иванов → отдел → завод»

Пример для метрики «закрытые задачи в срок»:

1. **Личный KPI Иванова** = факт Иванова / план Иванова.
2. **KPI службы главного инженера** = агрегат по всем сотрудникам службы (взвешенно или суммой).
3. **Метрика завода в дашборде** = агрегат по всем отделам.

Ключевой принцип: одна и та же метрика на разных уровнях должна считаться по формально описанному правилу (`aggregation_type` + `weight`).

## 5) Как хранить KPI-логику в GitHub (как «репозиторий KPI»)

Можно сделать репозиторий как «конфиг + код»:

```text
kpi-repo/
  configs/
    metrics.yml
    employees.yml
    departments.yml
    targets/
      2026-Q1.yml
  transformations/
    dbt_models/
  etl/
    bitrix24_loader.py
  tests/
    test_metric_rules.py
  docs/
    kpi_catalog.md
```

Идея: правила KPI (веса, цели, агрегации) меняются через pull request, с ревью и историей изменений.

## 6) Поэтапный план внедрения

### Этап 1 — MVP (быстро)
- 10–20 ключевых метрик.
- 2–3 отдела + 1 заводской дашборд.
- Ежедневное обновление.
- Ручная валидация расчетов с владельцами функций.

### Этап 2 — Стабилизация
- Подключение всех отделов.
- Автотесты качества данных:
  - нет ли пустых `employee_id` там, где обязательны;
  - нет ли дублей;
  - контроль аномалий по отклонению.
- SLA по времени обновления.

### Этап 3 — Масштабирование
- Переход на оркестрацию (Airflow/Prefect).
- Инкрементальные загрузки вместо полных.
- Историзация оргструктуры (SCD2), чтобы KPI считались корректно при переводах сотрудников.
- Ролевая модель доступа (руководитель видит свой контур).

---

## 7) Что выбрать прямо сейчас

Если приоритет — **максимально быстро**:
- берите **Вариант A** (n8n + таблицы + BI),
- но сразу ведите `metrics.yml` и `targets` в GitHub, чтобы перейти к варианту B без боли.

Если приоритет — **не переделывать через 3 месяца**:
- начинайте с **Варианта B** (PostgreSQL + dbt + BI),
- это лучше для 1000+ полей и большого числа сотрудников.

---

## 8) Практический стартовый чек-лист

1. Зафиксировать каталог метрик (владелец, формула, периодичность, уровень).
2. Зафиксировать оргструктуру (сотрудник → отдел → завод).
3. Определить единый идентификатор сотрудника и отдела во всех источниках.
4. Описать 2–3 типа агрегации (sum/ratio/weighted).
5. Запустить первый сквозной сценарий на одной метрике.
6. Сверить результаты вручную и только потом масштабировать на 1000 полей.
